üîå REST API Design

Best practices and patterns for building RESTful APIs

üìê Core Design Principles
Client-Server Architecture
Separation of concerns between UI and data storage
Stateless
Each request contains all information needed to process it
Cacheable
Responses must define themselves as cacheable or not
Uniform Interface
Consistent resource identification and manipulation
Layered System
Client cannot tell if connected directly to end server
Code on Demand (Optional)
Servers can extend client functionality by transferring code
REST: Representational State Transfer - architectural style for distributed systems.
üéØ Resource Naming Conventions
Use nouns, not verbs
/users not /getUsers or /createUser
Use plural nouns
/users/123 not /user/123
Use lowercase and hyphens
/user-profiles not /userProfiles or /user_profiles
Hierarchical relationships
/users/123/orders/456 for nested resources
Avoid deep nesting
Limit to 2-3 levels: /resources/{id}/sub-resources
Use query parameters for filtering
/users?status=active&role=admin
üí° Best practice: Keep URLs simple, predictable, and human-readable.
üîß HTTP Methods Usage
GET /resources
Retrieve list of resources (collection)
GET /resources/{id}
Retrieve specific resource by ID
POST /resources
Create new resource, return 201 with Location header
PUT /resources/{id}
Replace entire resource (full update), idempotent
PATCH /resources/{id}
Partial update of resource, idempotent
DELETE /resources/{id}
Remove resource, return 204 No Content
HEAD /resources/{id}
Check if resource exists without retrieving body
OPTIONS /resources
Discover allowed methods for resource
Idempotent: Multiple identical requests have same effect as single request (GET, PUT, DELETE, HEAD, OPTIONS).
üìä Status Code Guidelines
200 OK
Successful GET, PUT, PATCH with response body
201 Created
Successful POST, include Location header
204 No Content
Successful DELETE or PUT without response body
400 Bad Request
Malformed request syntax or invalid JSON
401 Unauthorized
Authentication required or failed
403 Forbidden
Authenticated but not authorized
404 Not Found
Resource does not exist
409 Conflict
Request conflicts with current state
422 Unprocessable Entity
Valid syntax but failed validation
429 Too Many Requests
Rate limit exceeded
500 Internal Server Error
Unexpected server error
503 Service Unavailable
Server temporarily unavailable
üî¢ API Versioning Strategies
URL Path Versioning
/api/v1/users or /v1/users (most common)
Query Parameter Versioning
/api/users?version=1 (less common)
Header Versioning
Accept: application/vnd.api.v1+json
Custom Header Versioning
X-API-Version: 1 or API-Version: 1
Content Negotiation
Accept: application/json; version=1
üí° Recommendation: URL path versioning is most visible and easiest to use. Version only when breaking changes occur.
Semantic versioning: Use v1, v2, v3 for major versions. Avoid v1.1, v1.2 in URLs.
üìÑ Pagination Patterns
Offset-based Pagination
/users?limit=20&offset=40 (page 3)
Page-based Pagination
/users?page=3&per_page=20 (more intuitive)
Cursor-based Pagination
/users?cursor=eyJpZCI6MTIzfQ&limit=20 (best for real-time)
Response Metadata
Include total, page, per_page, total_pages in response
Link Headers
Link: </users?page=4>; rel="next", </users?page=2>; rel="prev"
Cursor pagination: Best for large datasets and real-time data. Prevents missing/duplicate items when data changes.
üîç Filtering, Sorting & Searching
Simple Filtering
/users?status=active&role=admin
Range Filtering
/products?price_min=10&price_max=100
Date Range Filtering
/orders?created_after=2024-01-01&created_before=2024-12-31
Sorting
/users?sort=created_at or /users?sort=-created_at (descending)
Multiple Sort Fields
/users?sort=last_name,first_name or sort[]=last_name&sort[]=first_name
Full-text Search
/users?q=john or /search?query=john&type=users
Field Selection
/users?fields=id,name,email (sparse fieldsets)
üí° Convention: Use minus (-) prefix for descending sort: ?sort=-created_at
üîê Authentication Patterns
Bearer Token (JWT)
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
API Key in Header
X-API-Key: your-api-key-here
API Key in Query
/api/users?api_key=your-key (less secure, avoid if possible)
OAuth 2.0
Authorization: Bearer {access_token} (industry standard)
Basic Authentication
Authorization: Basic base64(username:password)
‚ö†Ô∏è Security: Always use HTTPS. Never send credentials in query parameters. Implement token expiration and refresh.
‚ö° Rate Limiting
X-RateLimit-Limit
Maximum requests allowed in time window
X-RateLimit-Remaining
Number of requests remaining in current window
X-RateLimit-Reset
Unix timestamp when rate limit resets
429 Too Many Requests
Return when limit exceeded with Retry-After header
Retry-After
Seconds to wait before retrying (e.g., Retry-After: 60)
Common limits: 1000 requests/hour for authenticated, 100/hour for unauthenticated users.
‚ùå Error Response Format
Consistent Error Structure
{ "error": { "code": "VALIDATION_ERROR", "message": "...", "details": [...] } }
Error Code
Machine-readable error identifier (e.g., INVALID_EMAIL)
Error Message
Human-readable description of the error
Error Details
Array of field-specific validation errors
Request ID
Include unique request ID for debugging and support
Documentation Link
URL to error documentation or troubleshooting guide
üí° Best practice: Never expose internal error details or stack traces in production.
üì¶ Response Format Best Practices
Consistent JSON Structure
Use same envelope format across all endpoints
camelCase for JSON
{ "firstName": "John", "lastName": "Doe" }
snake_case Alternative
{ "first_name": "John", "last_name": "Doe" } (Python/Ruby style)
ISO 8601 for Dates
2024-11-29T10:30:00Z or 2024-11-29T10:30:00+00:00
Null vs Omission
Include null for known empty values, omit for unknown/not applicable
Metadata Wrapper
{ "data": [...], "meta": { "total": 100, "page": 1 } }
HATEOAS Links
Include links to related resources: { "links": { "self": "/users/123" } }
Consistency: Choose one naming convention (camelCase or snake_case) and use it everywhere.
üîó HATEOAS (Hypermedia)
Self Link
"_links": { "self": { "href": "/users/123" } }
Related Resources
"_links": { "orders": { "href": "/users/123/orders" } }
Actions
"_links": { "activate": { "href": "/users/123/activate", "method": "POST" } }
Pagination Links
"_links": { "next": "/users?page=2", "prev": "/users?page=1" }
HATEOAS: Hypermedia as the Engine of Application State - clients discover actions through links.

üõ°Ô∏è Security Best Practices
Always use HTTPS
Encrypt all API traffic with TLS/SSL
Implement authentication
Require authentication for all non-public endpoints
Use authorization
Verify user permissions for each resource access
Validate all input
Sanitize and validate request data to prevent injection attacks
Implement rate limiting
Prevent abuse and DDoS attacks
Use CORS properly
Configure Access-Control-Allow-Origin carefully
Don't expose sensitive data
Never return passwords, tokens, or internal IDs unnecessarily
Log security events
Track authentication failures, authorization denials
Use security headers
Implement HSTS, CSP, X-Content-Type-Options
‚ö†Ô∏è Critical: Never trust client input. Always validate and sanitize on the server side.
‚ö° Performance Best Practices
Implement caching
Use Cache-Control, ETag, and conditional requests
Enable compression
Use gzip or brotli for response compression
Paginate large collections
Never return unbounded lists
Use field selection
Allow clients to request only needed fields
Implement async operations
Return 202 Accepted for long-running tasks with status endpoint
Use connection pooling
Reuse database connections efficiently
Optimize database queries
Use indexes, avoid N+1 queries, implement eager loading
Monitor and log
Track response times, error rates, and usage patterns
üí° Performance tip: Use ETags for conditional requests to reduce bandwidth and server load.
üìö Documentation Best Practices
Use OpenAPI/Swagger
Document API with OpenAPI 3.0 specification
Provide examples
Include request/response examples for every endpoint
Document error codes
List all possible error codes and their meanings
Include authentication guide
Explain how to obtain and use API credentials
Provide SDKs/client libraries
Offer libraries for popular languages
Keep changelog
Document all API changes and deprecations
Interactive documentation
Allow developers to test endpoints directly from docs
Good documentation: Clear, complete, and up-to-date documentation is crucial for API adoption.
