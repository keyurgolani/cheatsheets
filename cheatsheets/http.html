üîµ 1xx Informational

100 Continue
Server received request headers, client should send body
101 Switching Protocols
Server switching protocols as requested by client
102 Processing
Server received and processing request (WebDAV)
103 Early Hints
Used to return headers before final response
Use case: Informational responses indicate the request was received and is being processed.

‚úÖ 2xx Success

200 OK
Request succeeded, response contains requested data
201 Created
Request succeeded, new resource created (POST/PUT)
202 Accepted
Request accepted for processing, not yet completed
204 No Content
Request succeeded, no content to return (DELETE)
206 Partial Content
Partial GET request fulfilled (range requests)
üí° Best practice: Use 201 for resource creation and include Location header with new resource URI.

üîÄ 3xx Redirection

301 Moved Permanently
Resource permanently moved to new URL
302 Found
Resource temporarily at different URL
303 See Other
Response found at different URL using GET
304 Not Modified
Cached version is still valid (conditional GET)
307 Temporary Redirect
Temporary redirect, preserve request method
308 Permanent Redirect
Permanent redirect, preserve request method
Difference: 301/302 may change POST to GET, 307/308 preserve the original method.

‚ùå 4xx Client Errors

400 Bad Request
Malformed request syntax or invalid parameters
401 Unauthorized
Authentication required or failed
403 Forbidden
Server understood but refuses to authorize
404 Not Found
Requested resource does not exist
405 Method Not Allowed
HTTP method not supported for this resource
408 Request Timeout
Server timed out waiting for request
409 Conflict
Request conflicts with current state (version mismatch)
410 Gone
Resource permanently deleted
413 Payload Too Large
Request entity exceeds server limits
415 Unsupported Media Type
Request payload format not supported
422 Unprocessable Entity
Syntax correct but semantically invalid (validation)
429 Too Many Requests
Rate limit exceeded

‚ö†Ô∏è 5xx Server Errors

500 Internal Server Error
Generic server error, something went wrong
501 Not Implemented
Server doesn't support requested functionality
502 Bad Gateway
Invalid response from upstream server
503 Service Unavailable
Server temporarily unavailable (maintenance/overload)
504 Gateway Timeout
Upstream server failed to respond in time
505 HTTP Version Not Supported
HTTP version in request not supported
‚ö†Ô∏è Important: 5xx errors indicate server-side issues. Client should not retry without changes.

üì§ HTTP Request Methods

GET
Retrieve resource, safe and idempotent
POST
Create resource or submit data, not idempotent
PUT
Replace entire resource, idempotent
PATCH
Partially update resource, idempotent
DELETE
Remove resource, idempotent
HEAD
Same as GET but without response body
OPTIONS
Describe communication options for resource
TRACE
Echo request for debugging (rarely used)
CONNECT
Establish tunnel to server (for HTTPS proxies)
Idempotent: Multiple identical requests have same effect as single request.

üìã Common Request Headers

Accept
Media types client can process (e.g., application/json)
Authorization
Credentials for authentication (Bearer token, Basic auth)
Content-Type
Media type of request body (e.g., application/json)
Content-Length
Size of request body in bytes
User-Agent
Client application identifier
Cookie
HTTP cookies previously sent by server
Referer
URL of previous page that linked to current request
If-None-Match
ETag for conditional requests (caching)
If-Modified-Since
Date for conditional requests (caching)
Origin
Origin of request (for CORS)

üì® Common Response Headers

Content-Type
Media type of response body
Content-Length
Size of response body in bytes
Cache-Control
Caching directives (max-age, no-cache, public, private)
ETag
Unique identifier for resource version
Location
URL for redirects or newly created resources
Set-Cookie
Send cookie to client
Access-Control-Allow-Origin
CORS: allowed origins
Access-Control-Allow-Methods
CORS: allowed HTTP methods
Access-Control-Allow-Headers
CORS: allowed request headers
Retry-After
When to retry after 503 or 429

üîí Security Headers

Strict-Transport-Security
Force HTTPS (HSTS)
Content-Security-Policy
Control resource loading to prevent XSS
X-Content-Type-Options
Prevent MIME type sniffing (nosniff)
X-Frame-Options
Prevent clickjacking (DENY, SAMEORIGIN)
X-XSS-Protection
Enable browser XSS filtering
Referrer-Policy
Control referrer information
üí° Security best practice: Always implement security headers in production environments.

üîÑ Caching Headers

Cache-Control: no-cache
Validate with server before using cached copy
Cache-Control: no-store
Never cache this response
Cache-Control: public
Can be cached by any cache
Cache-Control: private
Only browser cache, not shared caches
Cache-Control: max-age=3600
Cache for 3600 seconds (1 hour)
Cache-Control: must-revalidate
Must validate stale cache with server
Expires
Absolute expiration date (legacy, use max-age)
Last-Modified
When resource was last modified
Caching strategy: Use ETag + If-None-Match for validation, max-age for freshness.

üí° Best practice: Return appropriate status codes with descriptive error messages in response body.

üí° HTTP Best Practices

Use appropriate status codes
Don't return 200 for errors or 500 for validation failures
Implement proper caching
Use Cache-Control, ETag, and conditional requests
Version your APIs
Use URL versioning (/v1/) or Accept header
Use HTTPS everywhere
Encrypt all traffic, implement HSTS
Implement rate limiting
Return 429 with Retry-After header
Handle CORS properly
Set appropriate Access-Control-* headers
Use compression
Enable gzip/brotli for text-based responses
Provide meaningful error messages
Include error details in response body
Use idempotent methods correctly
GET, PUT, DELETE should be idempotent
Include Location header
For 201 Created and 3xx redirects
