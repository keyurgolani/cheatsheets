Five fundamental principles for maintainable object-oriented design

S - Single Responsibility Principle
A class should have one, and only one, reason to change. Each class should have a single, well-defined responsibility.
‚ùå Violation Example
class User { constructor(public name: string, public email: string) {} // Responsibility 1: User data getName(): string { return this.name; } // Responsibility 2: Database operations save(): void { // Save to database } // Responsibility 3: Email notifications sendEmail(message: string): void { // Send email } }
‚úÖ Correct Implementation
class User { constructor(public name: string, public email: string) {} getName(): string { return this.name; } } class UserRepository { save(user: User): void { /* DB logic */ } } class EmailService { send(email: string, message: string): void { /* Email logic */ } }
Benefits:
‚Ä¢ Easier to understand and maintain
‚Ä¢ Changes to one responsibility don't affect others
‚Ä¢ Better testability
‚Ä¢ Improved code reusability
O - Open/Closed Principle
Software entities should be open for extension but closed for modification. You should be able to add new functionality without changing existing code.
‚ùå Violation Example
class AreaCalculator { calculate(shape: any): number { if (shape.type === 'circle') { return Math.PI * shape.radius ** 2; } else if (shape.type === 'rectangle') { return shape.width * shape.height; } // Adding new shape requires modifying this class } }
‚úÖ Correct Implementation
interface Shape { area(): number; } class Circle implements Shape { constructor(private radius: number) {} area(): number { return Math.PI * this.radius ** 2; } } class Rectangle implements Shape { constructor(private width: number, private height: number) {} area(): number { return this.width * this.height; } } class AreaCalculator { calculate(shape: Shape): number { return shape.area(); // No modification needed for new shapes } }
Benefits:
‚Ä¢ Reduces risk of breaking existing functionality
‚Ä¢ Promotes code reusability through inheritance/composition
‚Ä¢ Makes code more flexible and maintainable
‚Ä¢ Easier to add new features
L - Liskov Substitution Principle
Objects of a superclass should be replaceable with objects of a subclass without breaking the application. Subtypes must be substitutable for their base types.
‚ùå Violation Example
class Bird { fly(): void { console.log('Flying...'); } } class Penguin extends Bird { fly(): void { throw new Error('Penguins cannot fly!'); } } // This breaks when using Penguin as Bird function makeBirdFly(bird: Bird) { bird.fly(); // Throws error for Penguin! }
‚úÖ Correct Implementation
class Bird { move(): void { console.log('Moving...'); } } class FlyingBird extends Bird { fly(): void { console.log('Flying...'); } } class Penguin extends Bird { swim(): void { console.log('Swimming...'); } } // Now all birds can be used interchangeably function makeBirdMove(bird: Bird) { bird.move(); // Works for all birds }
Benefits:
‚Ä¢ Ensures correct inheritance hierarchies
‚Ä¢ Prevents unexpected behavior in polymorphic code
‚Ä¢ Makes code more predictable and reliable
‚Ä¢ Improves code reusability
I - Interface Segregation Principle
Clients should not be forced to depend on interfaces they don't use. Many specific interfaces are better than one general-purpose interface.
‚ùå Violation Example
interface Worker { work(): void; eat(): void; sleep(): void; } class HumanWorker implements Worker { work(): void { console.log('Working...'); } eat(): void { console.log('Eating...'); } sleep(): void { console.log('Sleeping...'); } } class RobotWorker implements Worker { work(): void { console.log('Working...'); } eat(): void { // Robots don't eat! } sleep(): void { // Robots don't sleep! } }
‚úÖ Correct Implementation
interface Workable { work(): void; } interface Eatable { eat(): void; } interface Sleepable { sleep(): void; } class HumanWorker implements Workable, Eatable, Sleepable { work(): void { console.log('Working...'); } eat(): void { console.log('Eating...'); } sleep(): void { console.log('Sleeping...'); } } class RobotWorker implements Workable { work(): void { console.log('Working...'); } }
Benefits:
‚Ä¢ Reduces coupling between classes
‚Ä¢ Prevents "fat" interfaces with unused methods
‚Ä¢ Makes code more flexible and maintainable
‚Ä¢ Easier to implement and test
D - Dependency Inversion Principle
High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details; details should depend on abstractions.
‚ùå Violation Example
class MySQLDatabase { save(data: string): void { console.log('Saving to MySQL...'); } } class UserService { private database = new MySQLDatabase(); saveUser(user: string): void { // Tightly coupled to MySQLDatabase this.database.save(user); } }
‚úÖ Correct Implementation
interface Database { save(data: string): void; } class MySQLDatabase implements Database { save(data: string): void { console.log('Saving to MySQL...'); } } class MongoDatabase implements Database { save(data: string): void { console.log('Saving to MongoDB...'); } } class UserService { constructor(private database: Database) {} saveUser(user: string): void { // Depends on abstraction, not concrete implementation this.database.save(user); } }
Benefits:
‚Ä¢ Reduces coupling between modules
‚Ä¢ Makes code more testable (easy to mock dependencies)
‚Ä¢ Allows easy swapping of implementations
‚Ä¢ Promotes flexibility and maintainability
üéØ Quick Reference Guide
S - Single Responsibility
One class = One job. If you can describe a class with "AND", it's doing too much.
O - Open/Closed
Add new features by extending, not modifying. Use interfaces and abstract classes.
L - Liskov Substitution
Subclasses should enhance, not replace, parent behavior. Don't break contracts.
I - Interface Segregation
Many small interfaces > One large interface. Don't force implementations of unused methods.
D - Dependency Inversion
Depend on abstractions, not concrete classes. Use dependency injection.
üí° Practical Application Tips
Start with Single Responsibility:
This is the foundation. If you get this right, other principles become easier to follow.
Use Dependency Injection:
Pass dependencies through constructors or setters rather than creating them inside classes. This naturally leads to better adherence to multiple SOLID principles.
Favor Composition Over Inheritance:
When in doubt, use composition. It's more flexible and helps avoid Liskov Substitution violations.
Design Interfaces Carefully:
Keep interfaces small and focused. It's easier to combine multiple interfaces than to split a large one later.
Refactor Gradually:
Don't try to apply all principles at once. Refactor incrementally as you identify code smells.
üö® Common Violations & Code Smells
God Classes: Classes that do everything. Usually violate Single Responsibility.
Switch/If-Else Chains: Type checking and branching logic. Often violate Open/Closed.
Refused Bequest: Subclasses that don't use inherited methods. Violates Liskov Substitution.
Fat Interfaces: Interfaces with many methods that clients don't need. Violates Interface Segregation.
Tight Coupling: Classes that directly instantiate their dependencies. Violates Dependency Inversion.
Remember: SOLID principles are guidelines, not strict rules. Apply them pragmatically based on your specific context and requirements.
üìö Benefits of Following SOLID
Maintainability: Code is easier to understand, modify, and extend over time.
Testability: Loosely coupled code with clear responsibilities is easier to test.
Flexibility: Changes in requirements can be accommodated with minimal code changes.
Reusability: Well-designed components can be reused across different parts of the application.
Scalability: Systems built on SOLID principles scale better as complexity grows.
The Bottom Line: SOLID principles help you write code that's easier to understand, maintain, and extend. They reduce technical debt and make your codebase more resilient to change.
