üîÑ Common Troubleshooting Workflows
Basic Connectivity Issue
Check physical connection (cable, WiFi)
Verify interface is up: ip addr show or ifconfig
Ping localhost: ping 127.0.0.1 (tests TCP/IP stack)
Ping gateway: ping [gateway_ip] (tests local network)
Ping external IP: ping 8.8.8.8 (tests internet connectivity)
Ping domain: ping google.com (tests DNS resolution)
If domain fails but IP works, check DNS: cat /etc/resolv.conf
Slow Connection Diagnosis
Check packet loss: ping -c 100 example.com
Identify problematic hop: mtr example.com
Check interface errors: netstat -i or ip -s link
Test bandwidth: iperf3 -c server
Check for congestion: ss -s (socket statistics)
Monitor real-time traffic: iftop or nethogs
DNS Resolution Problems
Test with different DNS: dig @8.8.8.8 example.com
Check local DNS config: cat /etc/resolv.conf
Flush DNS cache (varies by OS)
Trace DNS path: dig +trace example.com
Check authoritative servers: dig NS example.com
Verify DNS propagation: Query multiple DNS servers
Port/Service Not Accessible
Verify service is running: systemctl status service_name
Check if port is listening: ss -tuln | grep :port
Test local connectivity: telnet localhost port
Test remote connectivity: telnet remote_ip port
Check firewall rules: iptables -L or firewall-cmd --list-all
Capture packets: sudo tcpdump -i any port [port]
Check for connection attempts in logs
High Latency Investigation
Baseline latency: ping -c 100 target
Identify where delay occurs: mtr -r target
Check local interface: ethtool eth0
Monitor CPU/memory on both ends
Check for network congestion: iftop
Test different times of day (ISP throttling?)
Compare with alternative route/VPN
Packet Loss Analysis
Measure loss rate: ping -c 1000 target
Identify where loss occurs: mtr -r -c 100 target
Check interface statistics: ip -s link show eth0
Look for errors/drops in interface stats
Check physical layer (cable, WiFi signal)
Test with different packet sizes: ping -s 1400 target
Capture and analyze: sudo tcpdump -i any -w loss.pcap
üéØ tcpdump Filter Examples
sudo tcpdump 'tcp[tcpflags] & tcp-syn != 0'
Capture SYN packets (connection attempts)
sudo tcpdump 'src host 192.168.1.1'
Capture from specific source
sudo tcpdump 'dst port 443'
Capture to destination port 443
sudo tcpdump 'net 192.168.1.0/24'

sudo tcpdump 'icmp'

sudo tcpdump 'not port 22'

sudo tcpdump 'portrange 80-443'
Capture port range
sudo tcpdump 'greater 1000'
Capture packets larger than 1000 bytes
üí° Tip: Combine filters with 'and', 'or', 'not' for complex captures
‚ö†Ô∏è Common Network Issues
Issue: "Destination Host Unreachable"
Cause: No route to host, host is down, or firewall blocking
Check: Routing table, host status, firewall rules
Fix: Add route, start host, adjust firewall
Issue: "Connection Timed Out"
Cause: Firewall blocking, service not listening, network congestion
Check: Service status, firewall, packet capture
Fix: Start service, open firewall port, check network path
Issue: "Connection Refused"
Cause: Service not running or not listening on that port
Check: Service status, listening ports
Fix: Start service, verify configuration
Issue: "Network Unreachable"
Cause: No default gateway or incorrect routing
Check: Routing table, gateway configuration
Fix: Add default route, configure gateway
Issue: High Packet Loss
Cause: Bad cable, WiFi interference, congestion, faulty hardware
Check: Physical connections, interface errors, mtr output
Fix: Replace cable, change WiFi channel, upgrade hardware
Issue: DNS Not Resolving
Cause: DNS server down, incorrect configuration, firewall blocking port 53
Check: /etc/resolv.conf, DNS server reachability, firewall
Fix: Use alternative DNS (8.8.8.8), fix configuration, open port 53
üí° Best Practices
Document Your Baseline
Know your normal latency, bandwidth, and packet loss rates. This makes anomalies easier to spot.
Work from Layer 1 Up
Start with physical layer (cables, WiFi), then network layer (IP, routing), then application layer.
Use Multiple Tools
Confirm findings with different tools. ping + mtr + traceroute gives better picture than any single tool.
Capture Before Making Changes
Use tcpdump to capture traffic before troubleshooting. You can analyze it later if needed.
Check Both Ends
Network issues can be on either side. Test from both client and server perspectives.
Security Considerations
‚Ä¢ tcpdump and Wireshark can capture sensitive data - use on trusted networks only
‚Ä¢ Some tools require root/admin privileges
‚Ä¢ Port scanning (nmap) may trigger security alerts
‚Ä¢ Always get permission before troubleshooting production systems
